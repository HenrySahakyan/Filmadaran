# Videodaran — Кинематографичный Гид по Сериалам

Videodaran — это Android-приложение, созданное с использованием **многомодульной архитектуры** и принципов **Clean Architecture**. Этот проект — не просто реализация списка и деталей, а демонстрация системы, которая готова к росту и поддержке в условиях реального Enterprise-проекта.

## 🏗 Архитектурные решения: Почему именно так?

При разработке я руководствовался принципом "Clean Code", чтобы приложение оставалось гибким даже при десятикратном увеличении кодовой базы.

### 1. Многомодульность (Задел на будущее)
Проект разделен на слои: `:domain`, `:presentation`, `:data` и `:core`. 
*   **Масштабируемость**: В больших командах это единственный способ избежать конфликтов при слиянии веток. Каждый модуль имеет свою зону ответственности.
*   **Будущий рост**: Если завтра мы решим добавить «Личный кабинет» или «ТВ-программу», мы просто создадим новый модуль (например, `:feature:profile`). Это гарантирует быструю сборку, так как Gradle будет пересобирать только те части, которые изменились.
*   **Изоляция**: Компилятор физически не позволит слою UI обращаться к базе данных напрямую. Весь обмен данными идет через Domain-слой.

### 2. Clean Architecture + Use Cases
Вместо того чтобы ViewModels напрямую работали с репозиториями, я выделил бизнес-логику в отдельные классы — **Use Cases (Интерфейсы взаимодействия)**.
*   **Обоснование**: Это делает логику тестируемой и переиспользуемой. Например, `ToggleFavoriteUseCase` можно использовать на любом экране без дублирования кода.

### 3. Почему Koin, а не Dagger/Hilt?
Для внедрения зависимостей я выбрал **Koin**.
*   **Производительность**: Dagger использует тяжелую генерацию кода (KAPT/KSP), что замедляет сборку. Koin — это чистый Kotlin DSL. Он работает быстро, не требует кодогенерации и его намного проще читать.
*   **Лаконичность**: Функции `viewModelOf` и `singleOf` позволяют избавиться от «бойлерплейта», сохраняя при этом все преимущества гибкого DI.

### 4. Работа с данными
*   **Persistence**: Я внедрил **Room** для хранения избранного. В серьезных приложениях данные пользователя не должны пропадать при перезапуске.
*   **Навигация через ID**: При переходе на экран деталей передается только ID, и данные подгружаются заново. Это гарантирует актуальность информации и позволяет легко реализовать Deep Links.

## 📦 Технологический стек
*   **Kotlin** & **MVVM**
*   **Retrofit** & **Gson** для сетевых запросов
*   **Room DB** для локального хранения
*   **Flow & Coroutines** для реактивной обработки данных
*   **Material 3** с поддержкой Edge-to-Edge дизайна
